# フリーランス案件探しがバラバラすぎたので、横断検索ダッシュボード「fl-cross」を作った

> 複数のフリーランスエージェント／クラウドソーシングを **1画面で横断**し、
> 自分のスキルに対する **マッチ率（適合度）** を出して案件を絞り込むツールを作りました。
> Streamlit + Playwright + ローカルLLM(Ollama) の構成です。
> GitHub: https://github.com/zephel01/fl-cross

---

## TL;DR

- レバテック／フリーランスHub／Findy／クラウドワークステック／ランサーズ／フリーランスボードを横断
- スキル・単価・リモート・稼働日数からマッチ率を自動算出して並べ替え
- リモート区分／稼働日数（週何日）／47都道府県／提供元除外 などで細かく絞り込み
- SPA・ログイン必須サイトは Playwright（永続プロファイルでログイン保持）で取得
- 気になる案件はローカルLLMで深掘り分析（プライバシーを担保）

---

## なぜ作ったのか

フリーランスの案件探しをしていると、とにかく **探し先がバラバラ**で消耗します。

- レバテックやFindyのようなエージェント型
- クラウドワークステックやランサーズのようなクラウドソーシング型
- フリーランスHubやフリーランスボードのようなアグリゲーター型

それぞれUIも検索条件もバラバラ。同じキーワードを何度も入れ直し、単価表記も「月額」「時給」「〜万円」と統一されておらず、**横断で比較できない**。さらに「自分のスキルにどれくらい合うのか」は結局、毎回目視で判断していました。

「だったら、取得→正規化→採点→一覧表示までを1画面に共通化すればいいのでは？」というのが出発点です。

---

## fl-cross とは

ひとことで言うと **「フリーランス案件の横断検索＋マッチ率ダッシュボード」** です。

各サイトから取得した案件を統一スキーマに正規化し、自分のプロフィール（強み・希望単価・働き方）に対するマッチ率を出して、降順で並べます。

> 📷 **画像①：ダッシュボード全体**
> （`docs/images/01_dashboard.png` — マッチ率・単価つきの案件一覧。左にサイドバー）

折りたたんだ一覧でも「マッチ率 ｜ 💰単価 ｜ タイトル ｜ 取得元/提供元」が一目で分かるようにしています。色（🟢🟡⚪）でマッチ率帯も即判断できます。

## 主な機能

### 1. 横断検索＋マッチ率

案件を開くと、マッチ率の内訳（キーワード一致 / 単価 / リモート / 新着度）や、単価・働き方・リモート区分・稼働日数・エリアといったメタ情報が確認できます。

> 📷 **画像②：案件カードを展開**
> （`docs/images/02_job_expanded.png` — スコア内訳とメタ情報、LLM深掘りボタン）

### 2. 細かいフィルタ

「フルリモートで週3日、関東、月80万以上」のような現実的な条件でサッと絞れるようにしました。

- リモート区分（フルリモート／一部リモート／リモート／常駐）
- 稼働日数（週1〜5。範囲案件にも対応：週3を選ぶと「週3〜5」もヒット）
- エリア（47都道府県）
- 最低時給
- **提供元（provider）除外** ← これが地味に効きます（後述）

> 📷 **画像③：サイドバーのフィルタ群**
> （`docs/images/03_sidebar.png`）

### 3. データ取得（3系統）

- **httpx自動取得（軽量）**：サーバー描画サイト（ランサーズ等）はブラウザ不要で取得
- **Playwright自動取得**：SPA・ログイン必須サイトも実ブラウザで描画して取得
- **手動取り込み**：任意の案件をJSON貼り付け or 1件追加

> 📷 **画像④：取り込みタブ**
> （`docs/images/04_import.png`）

### 4. ローカルLLMで深掘り

気になった案件だけ、ローカルのOllamaで「適合度の内訳・想定単価の妥当性・レッドフラッグ・提案ドラフト」まで分析。案件内容を外部に送らないので安心です。Ollamaが無くてもプロンプトをコピーして他のLLMに投げられます。

## 技術構成と工夫

### 構成

- **UI**：Streamlit（1ファイルのダッシュボード）
- **取得**：httpx（軽量）＋ Playwright（SPA/ログイン）
- **採点**：ルールベース ＋ ローカルLLM（Ollama）
- **データ**：統一 `Job` スキーマ → `data/jobs.json` に蓄積・重複排除

`core/` をスキーマ・取得・正規化・採点・エリア判定・稼働日数判定などに分割し、サイト追加や調整をしやすくしています。

### 壁①：SPAとログイン

最初は httpx だけで取ろうとしましたが、すぐ壁にぶつかりました。

- フリーランスHub・クラウドワークステックは **完全クライアント描画（SPA）** で、httpxではHTMLの“ガワ”しか取れない
- レバテック・Findyは **ログイン必須**

そこで **Playwright** を導入。実ブラウザで描画させ、ログイン情報は **永続プロファイル**に保存して再利用する設計にしました（パスワードはアプリに持たせない）。初回だけ `python fetch_browser.py --login` でログインすれば、以後はワンボタンで全サイト取得できます。

実測した「自動取得の可否」はこんな感じです：

| サイト | ブラウザ取得 | httpx軽量取得 |
|---|:---:|:---:|
| ランサーズ | ✅ | ✅ |
| フリーランスボード | ✅ | ✅ |
| フリーランスHub | ✅ | ❌（SPA） |
| クラウドワークステック | ✅ | ❌（CSR） |
| レバテック | ✅（要ログイン） | ❌ |
| Findy | ✅（要ログイン） | ❌ |

### 壁②：単価・条件表記の正規化

「〜120万円/月」「9,000円/時」「週3日〜」「フルリモート」など、サイトごとに表記がバラバラ。正規表現で**時給／月額のレンジ**、**リモート区分**、**稼働日数（範囲対応）**、**都道府県**を推定し、統一スキーマに落とし込みました。

### 工夫：ハイブリッド採点

マッチ率は **ルール採点（即時・無料）** を基本にしています：
キーワード一致 ＋ 単価適合 ＋ リモート/働き方 ＋ 新着度を重み付け。重みは設定ファイルで調整可能。
そのうえで、**選んだ案件だけ** ローカルLLMで深掘りする二段構えにして、速度とコストとのバランスを取りました。

### 気づき：アグリゲーター経由の“混入”

「レバテックを取得対象から外したのに、レバテックの案件が一覧に残る」という現象に遭遇。原因は、アグリゲーター（フリーランスHub）経由で**提供元がレバテックの案件**が入ってくるためでした。
そこで「取得元サイトのON/OFF」とは別軸で、**提供元（provider）単位の横断除外**を追加。表記揺れ（例「ココナラテック（旧：フリ…」）も代表名に正規化しています。

## 使い方（ざっくり）

```bash
git clone https://github.com/zephel01/fl-cross
cd fl-cross
pip install -r requirements.txt
playwright install chromium          # 全サイト取得を使う場合
cp config.example.toml config.toml   # 自分のスキル・希望条件を編集
streamlit run app.py
```

ログイン必須サイトを使うときは初回のみ：

```bash
python fetch_browser.py --login      # Findy/レバテックにログイン → 保存
```

## おわりに

「探し先がバラバラで面倒」という個人的な不満から始めましたが、横断・正規化・採点を1画面に集約したことで、案件比較がかなり楽になりました。

スクレイピングは各サイトの規約・robots.txtに配慮し、過度なアクセスは避ける前提です。マッチ率やLLM分析はあくまで参考情報なので、最終判断はご自身で。
